# 一、引言

## 1.1 标识

**系统名称**：江西小炒明档点菜与后厨协同系统

**英文缩写**：JXMC-OCKS

**版本号**：V1.0

**文档发行号**：FAR-20260425

**文档编制日期**：2026年4月25日

## 1.2 背景

本项目由江西小炒饭店实际运营需求提出。饭店采用“看菜点菜”模式（客人指冰柜内食材点菜），每日食材不固定。当前使用纸质菜单，前台手写后送至后厨，导致漏菜、上菜不及时、沟通效率低，且后厨厨师不识字，无法依靠文字菜单。项目目标是开发一套基于平板电脑的点菜与后厨协同系统，提高点菜效率，减少差错，并适应动态食材和厨师特殊需求。

实现环境：店内Wi-Fi网络覆盖、平板电脑一台、服务器.

## 1.3 项目概述

**用途**：实现明档点菜（图片展示）、后厨实时跟踪、动态价格调整、结账及订单导出。

**特性**：以图片为主的交互界面，支持手动输入临时菜品，后厨按时间顺序显示待做菜品，厨师点击完成标记。

**历史**：全新开发，无已有系统。

**相关方**：

投资方/需方/用户：江西小炒饭店（家庭式经营）

支持机构：学校

**运行现场**：饭店内（服务器固定放置，平板移动使用）

**其他文档**：需求规格说明书、概要设计说明书、详细设计说明书、测试计划与报告。

# 三、可行性分析的前提

## 3.1 项目的要求

**功能要求**：点菜（含图片展示、手动输入临时菜品、添加备注、修改份数、中途加菜）、后厨跟踪（按时间顺序显示、完成标记、重做功能）、结账（显示明细、记录实收金额、修改桌号）、价格管理（动态修改单个菜价、批量调整价格）、数据导出（TXT格式）。

**性能要求**：页面加载时间<2秒，后厨页面3秒自动刷新。

**接口要求**：平板浏览器通过HTTP协议访问服务器。

**可靠性**：每提交一桌订单自动保存，防止数据丢失。

## 3.2 项目的目标

消除漏菜、上菜不及时问题。

实现不识字厨师看图做菜。

降低前台与后厨沟通成本。

提供简单结账和营业数据导出功能。

## 3.3 项目的环境、条件、假定和限制

**运行环境**：店内Wi-Fi覆盖，一台可24小时开机的服务器（旧安卓手机或笔记本），一台平板电脑。

**开发条件**：Python 3.8+，Flask框架，JSON文件存储，VS Code编辑器。

**假定**：用户（服务员、厨师）愿意接受平板操作，厨师认识数字和简单图标。

**限制**：开发周期短；开发人员无Web开发经验，需边学边做；不涉及硬件打印机或支付终端。

## 3.4 进行可行性分析的方法

采用实地观察、访谈（饭店老板/厨师）、技术调研、成本估算及对比分析法。

# 四、可选的方案

## 4.1 原有方案的优缺点、局限性及存在的问题

**原有方案**：纸质菜单，前台手写单子，后厨按单做菜。

**优点**：零成本，无需设备。

**缺点**：易漏单、字迹不清、无法记录时间、无法追溯责任；厨师不识字需他人念单；加菜、催菜难以管理。

**局限性**：无法适应动态食材和高效协同。

## 4.2 可重用的系统，与要求之间的差距

市面常见扫码点餐系统（如美团、客如云）基于固定菜单，不支持每日食材变化、不支持厨师图片界面、无法满足“看菜点菜”模式，且价格昂贵（年使用费约2000-5000元）。本系统需全新开发。

## 4.3 可选择的系统方案1：B/S架构 + 旧手机服务器

**描述**：平板浏览器访问Flask Web应用，服务器部署在店内旧安卓手机（通过Termux运行Python）。

**优势**：成本极低（利用闲置设备），维护简单，满足移动使用。

**劣势**：需学习Termux基本操作，手机性能可能影响并发（但店内最多7桌，无并发问题）。

## 4.4 可选择的系统方案2：B/S架构 + 笔记本电脑服务器

**描述**：使用笔记本作为服务器，平板访问。

**优势**：开发调试方便，无需额外配置。

**劣势**：笔记本不便长期留在店内，影响日常使用；功耗较高。

## 4.5 选择最终方案的准则

成本最低

易于部署和维护

不影响开发人员个人设备使用

满足7桌以内的性能需求

**最终选择方案1**（旧手机服务器），但开发测试阶段使用笔记本电脑，最终可无缝迁移。

# 五、所建议的系统

## 5.1 对所建议的系统的说明

系统采用B/S架构，后端使用Python Flask框架，前端采用HTML/CSS/JavaScript，数据存储于JSON文件。支持菜品图片库（约50张）、手动输入临时菜、添加备注、修改份数、动态价格修改。后厨页面自动刷新，完成标记后保留记录。结账支持修改桌号、记录实收金额、导出TXT格式订单。

## 5.2 数据流程和处理流程

**服务员点菜流程**：选择桌号 → 浏览图片菜品或手动输入临时菜品 → 添加至临时订单（支持暂存） → 提交整桌订单 → 保存到orders.json文件。

**后厨处理流程**：读取未结账订单 → 提取未完成菜品 → 按时间排序显示 → 厨师点击“完成”按钮 → 更新该菜品状态为已完成。

**结账流程**：选择桌号 → 显示订单明细 → 输入实付金额 → 标记订单为已结账 → 导出当日已结账订单。

## 5.3 与原系统的比较

**信息传递**：纸质手写 → 数字图片+文字

**差错率**：较高 → 极低

**厨师依赖**：需他人念单 → 直接看图

**数据追溯**：无 → 完整记录点菜、完成时间

## 5.4 影响(或要求)

### 5.4.1 设备

服务器：旧安卓手机（Android 6.0以上，支持Wi-Fi）或临时用笔记本电脑

客户端：平板电脑一台（或手机），分辨率不低于1024×600

网络：店内无线路由器

### 5.4.2 软件

服务器：Termux（Android终端）、Python 3、Flask框架、Pillow库（可选，用于图片处理）

客户端：任意现代浏览器（Chrome/Safari）

### 5.4.3 运行

服务员需掌握平板基本操作（点击图片、输入文字）。

厨师需学会点击“完成”按钮。

### 5.4.4 开发

开发工具：VS Code、Git（版本控制）

开发语言：Python、HTML、CSS、JavaScript

### 5.4.5 环境

室内常温，平板需防油污（可加保护套）。

### 5.4.6 经费

硬件：若已有旧手机和平板，则0元；若无平板，最低约500元购买二手平板。

软件：全部免费。

其他：无。

### 5.5 局限性

不能脱离Wi-Fi网络使用。

不支持多平板的实时并发（但店内场景只需一台平板轮流使用）。

无打印机小票输出（可通过导出TXT文件替代）。

结账不自动计算优惠抹零，需人工输入实收金额。
