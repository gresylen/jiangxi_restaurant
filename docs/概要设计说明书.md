## 一、 系统架构设计

### 1.1 系统整体架构

- **架构模式**：客户端-服务器架构，采用单页面应用（SPA）模式
- **服务器职责**：处理业务逻辑、数据存储与管理、API接口提供
- **客户端职责**：用户界面展示、用户交互处理、本地数据缓存
- **离线操作实现**：使用localStorage存储临时数据，网络恢复后自动同步

### 1.2 技术栈选择

- **后端**：Python 3.x + Flask + Flask-CORS（处理跨域请求）
- **前端**：HTML5 + CSS3 + JavaScript + jQuery（轻量级选择）
- **数据存储**：JSON文件（主要数据）+ 图片文件（菜品图片）
- **离线存储**：localStorage（临时数据）

## 二、 模块划分与功能

### 2.1 模块划分

| 模块名称 | 主要功能 | 依赖关系 |
|---------|---------|---------|
| 点菜模块 | 菜品展示、添加、修改、删除 | 依赖菜品数据 |
| 后厨模块 | 菜品状态管理、实时更新 | 依赖订单数据 |
| 结账模块 | 账单生成、支付记录、发票管理 | 依赖订单数据 |
| 管理模块 | 菜品管理、价格调整、数据备份、统计分析 | 依赖所有数据 |
| 数据模块 | 数据存储、读取、备份、恢复 | 被所有模块依赖 |
| 网络模块 | 数据同步、离线处理 | 被所有模块依赖 |

### 2.2 核心功能实现

- **点菜功能**：
  - 菜品展示：从JSON文件读取菜品数据，以图片网格形式展示
  - 添加菜品：点击菜品图片添加到订单，支持手动输入临时菜品
  - 修改菜品：修改菜品份数、备注
  - 删除菜品：删除已点菜品

- **后厨跟踪功能**：
  - 实时更新：使用AJAX轮询（每3秒）获取最新订单状态
  - 状态管理：支持菜品完成标记和重做操作
  - 排序规则：按时间顺序排序，相同菜品排在一起

- **结账功能**：
  - 账单生成：根据订单数据生成详细账单
  - 支付记录：记录实际支付金额
  - 发票管理：记录发票信息

- **数据管理功能**：
  - 菜品管理：添加新菜品、上传图片
  - 价格调整：修改单个菜品价格、批量调整价格
  - 数据导出：导出订单为TXT文件
  - 数据备份：定期自动备份数据
  - 统计分析：生成热销菜品、每日营业额统计

## 三、 数据结构设计

### 3.1 数据模型

- **订单数据结构**：
  ```json
  {
    "order_id": "20260421123456",
    "table_id": 1,
    "status": "未结账",
    "create_time": "2026-04-21 12:34:56",
    "items": [
      {
        "dish_id": 1,
        "dish_name": "红烧肉",
        "price": 38.0,
        "quantity": 1,
        "remark": "加辣",
        "status": "待做",
        "image_path": "images/dish1.jpg"
      }
    ],
    "total_amount": 38.0,
    "actual_payment": 40.0,
    "invoice_info": "公司名称：XXX有限公司，税号：1234567890"
  }
  ```

- **菜品数据结构**：
  ```json
  {
    "dish_id": 1,
    "dish_name": "红烧肉",
    "price": 38.0,
    "image_path": "images/dish1.jpg",
    "create_time": "2026-04-21 10:00:00"
  }
  ```

- **价格修改历史数据结构**：
  ```json
  {
    "dish_id": 1,
    "old_price": 35.0,
    "new_price": 38.0,
    "modify_time": "2026-04-21 10:00:00",
    "modify_by": "admin"
  }
  ```

- **统计数据结构**：
  ```json
  {
    "date": "2026-04-21",
    "total_orders": 10,
    "total_amount": 500.0,
    "hot_dishes": [
      {"dish_id": 1, "dish_name": "红烧肉", "quantity": 5},
      {"dish_id": 2, "dish_name": "鱼香肉丝", "quantity": 4}
    ]
  }
  ```

### 3.2 数据存储

- **JSON文件结构**：
  - `orders.json`：存储所有订单数据
  - `dishes.json`：存储所有菜品数据
  - `price_history.json`：存储价格修改历史
  - `statistics.json`：存储统计数据

- **图片文件存储**：
  - 存储路径：`static/images/dishes/`
  - 命名规则：`dish_{dish_id}.jpg`
  - 组织方式：按日期或类别创建子目录

- **数据备份策略**：
  - 自动备份：每天营业结束后自动备份所有数据文件
  - 备份路径：`backup/YYYYMMDD/`
  - 保留期限：保留最近7天的备份

## 四、接口设计

### 4.1 API接口

| 接口路径 | 方法 | 功能 | 参数 | 返回值 |
|---------|------|------|------|---------|
| `/api/dishes` | GET | 获取菜品列表 | 无 | 菜品列表JSON |
| `/api/dishes` | POST | 添加新菜品 | dish_name, price, image | 新菜品ID |
| `/api/dishes/{id}` | PUT | 修改菜品价格 | price | 成功/失败 |
| `/api/orders` | GET | 获取订单列表 | table_id (可选) | 订单列表JSON |
| `/api/orders` | POST | 创建新订单 | table_id, items | 订单ID |
| `/api/orders/{id}` | PUT | 更新订单 | items, status | 成功/失败 |
| `/api/orders/{id}` | DELETE | 删除订单 | 无 | 成功/失败 |
| `/api/orders/{id}/items` | POST | 添加菜品到订单 | dish_id, quantity, remark | 成功/失败 |
| `/api/orders/{id}/items/{item_id}` | DELETE | 从订单中删除菜品 | 无 | 成功/失败 |
| `/api/orders/{id}/status` | PUT | 更新订单状态 | status | 成功/失败 |
| `/api/orders/{id}/checkout` | POST | 结账 | actual_payment, invoice_info | 成功/失败 |
| `/api/export` | GET | 导出订单 | date (可选) | TXT文件 |
| `/api/statistics` | GET | 获取统计数据 | date (可选) | 统计数据JSON |
| `/api/backup` | POST | 手动备份数据 | 无 | 成功/失败 |

### 4.2 模块间接口

- **模块间通信**：通过本地函数调用和事件机制
- **数据传递方式**：
  - 同步调用：直接传递参数
  - 异步调用：使用回调函数或Promise

## 五、界面设计概览

### 5.1 页面布局

- **主要页面**：
  - 点菜页：顶部桌号选择，主体菜品图片网格，底部已点菜品列表
  - 后厨页：左侧待做菜品，右侧已完成菜品
  - 结账页：桌号选择，订单明细表格，支付信息输入
  - 管理页：价格管理、添加菜品、数据备份、统计分析四个子页面

- **导航方式**：底部导航栏，包含四个主要页面入口

- **响应式设计**：
  - 针对11英寸平板优化，同时支持其他尺寸平板和手机
  - 使用媒体查询和弹性布局

### 5.2 交互流程

- **点菜流程**：
  1. 选择桌号
  2. 浏览菜品图片，点击添加
  3. 修改菜品份数和备注
  4. 提交整桌订单

- **后厨操作流程**：
  1. 查看待做菜品列表
  2. 完成菜品后点击"完成"按钮
  3. 如需重做，点击已完成菜品的"重做"按钮

- **结账流程**：
  1. 选择桌号
  2. 查看订单明细
  3. 输入实际支付金额
  4. 输入发票信息（可选）
  5. 点击"结账"按钮

- **管理操作流程**：
  1. 进入管理页面
  2. 选择相应子页面（价格管理、添加菜品、数据备份、统计分析）
  3. 执行相应操作

## 六、部署与集成方案

### 6.1 部署环境

- **服务器部署**：
  - 在Windows笔记本（拯救者Y7000）上安装Python和Flask
  - 使用内置的Flask开发服务器（适合小型应用）
  - 配置端口为5000

- **客户端访问**：
  - 通过平板或手机的浏览器访问 `http://服务器IP:5000`
  - 建议将网站添加到主屏幕，实现类似APP的体验

- **网络环境要求**：
  - 店内Wi-Fi网络稳定
  - 带宽满足图片传输需求

### 6.2 集成方案

- **系统与现有设备集成**：
  - 利用现有平板或手机作为客户端
  - 利用现有Windows笔记本作为服务器

- **数据迁移方案**：
  - 系统初始化时创建默认菜品数据
  - 支持手动导入现有菜品数据
