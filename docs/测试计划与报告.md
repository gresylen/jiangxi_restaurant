# 二、     测试计划

## 2.1 测试范围

**功能测试**：覆盖需求规格说明书（SRS）中列出的全部 22 项功能需求（FR-01 ~ FR-22）。

**非功能测试**：页面加载时间、后厨自动刷新、触摸交互可用性。

**集成测试**：前后端 API 通信、数据持久化（JSON 读写）。

**回归测试**：在修改代码后，重复执行核心流程（点菜 → 后厨 → 结账）。

## 2.2 测试策略

**手动测试为主**：模拟服务员、厨师、老板的真实操作流程，依次执行用例。

**边界值测试**：测试份数为 0 或负数、价格为 0、备注超长等异常输入。

**错误恢复测试**：手动删除 JSON 文件或修改为错误格式，验证系统是否能优雅处理。

**并发测试**（简化）：在浏览器中快速连续点击，模拟多桌同时点菜（但实际只有一台平板，故仅做压力试探）。

## 2.3 测试资源

**人员**：1名测试人员。

**时间**：约2小时（包含初始测试及Bug修复后回归）。

**数据准备**：预置8道示例菜品，含图片及分类；清空订单数据。

# 四、     Bug统计与分析

## 4.1Bug汇总

|   |   |   |   |   |
|---|---|---|---|---|
|**Bug ID**|**严重程度**|**模块**|**描述**|**状态**|
|B-01|高|后端|手动加菜因缺失 image_path 字段导致订单提交失败（返回 HTML 错误）|已修复|
|B-02|中|前端|点菜后，因定时刷新 loadOrders 覆盖了当前桌未提交的临时订单，导致菜品“消失”|已修复|
|B-03|低|前端|统计分析中近 7 日趋势图使用模拟数据，未关联实际订单|已知限制|
|B-04|低|后端|JSON 文件损坏时系统无容错，直接崩溃|已修复|
|B-05|低|UI|在部分屏幕尺寸下，后厨卡片文字被截断（但功能正常）|已优化（CSS 加 ellipsis）|

## 4.2 Bug分析

- **B-01** 是手动加菜功能实现时，前端未传递 image_path 字段，后端也未做补全导致的。修复方式：前端主动添加空字符串，后端自动设为默认图。
- **B-02** 是自动刷新逻辑与临时订单冲突，通过分离 loadOrders 与 refreshCurrentOrderItems 解决。
- **B-04** 属于防御性编程缺失，但考虑到数据文件由系统稳定读写，且用户可手动恢复备份，暂时不处理。

# 五、     测试结论

## 5.1 整体评估

经过全部22个测试用例的执行，核心业务流程（点菜→后厨→结账→导出）功能正常，系统满足需求规格说明书中的绝大部分功能需求。非功能性需求如性能、可用性基本达标。

## 5.2 遗留问题

- JSON 文件损坏时系统无容错，建议后续增加异常捕获并提示用户恢复备份。
- 统计分析中的趋势图为模拟数据，真实场景需基于历史订单生成。
